# 蔵書管理システム 要件定義書

**作成日**: 2026-02-28
**ステータス**: ドラフト v0.4

---

## 1. 背景と課題

### 1.1 背景

Google Drive の `70_Book` フォルダ配下に、スキャン済み書籍 PDF 約 1,900 ファイルが保管されている。
ファイルはジャンル別サブフォルダに格納され、同一書籍について「オリジナル版 / iPad向け軽量版（ipad3_）/ Kindle向け軽量版（kindlep_）」の最大3バージョンが存在する。

書誌メタデータ（タイトル・著者・ISBN・ページ数・シリーズ名）はファイル名に独自形式で埋め込まれており、一元的な蔵書管理の仕組みが存在しない状態にある。

### 1.2 課題

| # | 課題 |
|---|------|
| 1 | 蔵書全体を横断的に検索・閲覧する手段がない |
| 2 | 同一書籍の複数バージョンを把握しにくい |
| 3 | ジャンル別・著者別の保有状況を俯瞰できない |
| 4 | 次に読む本を選ぶ際の指針が得られない |

---

## 2. プロジェクト目標

### 2.1 フェーズ1（本プロジェクト）

1. Google Drive のファイルリスト（CSV）からメタデータを自動解析し、正規化された蔵書カタログを構築する
2. 蔵書を検索・閲覧・フィルタリングできる仕組みを提供する
3. 書籍の詳細情報を外部データソースから補完して表示する

### 2.2 フェーズ2（将来）

- AI を活用した読書レコメンド機能

---

## 3. 実現方式

### 3.1 選択肢比較

| 方式 | 概要 | メリット | デメリット | 適合性 |
|------|------|---------|-----------|--------|
| **静的SPA** | React 等でビルドした HTML ファイルをブラウザで直接開く | サーバー不要、シンプル | 外部API呼び出し時にAPIキーが露出（公開APIは問題なし） | フェーズ1向き |
| **Webアプリ（フロントエンド + バックエンド）** | Node.js バックエンド + React フロントエンド、Docker 運用 | APIキーを安全に管理、フェーズ2への拡張がスムーズ | 構築・運用コストが高い | フェーズ2見越した構成 |
| **既存 Claude.ai Artifact** | 既存の JSX ファイルをそのまま利用 | 即時利用可 | Claude.ai 依存、拡張性が低い | 移行元 |

### 3.2 採用方式

**フェーズ1では静的SPA方式を採用する。** フェーズ2以降で必要に応じてバックエンドを追加する。

- 書籍詳細の補完情報取得（OpenBD / Google Books）はブラウザからの直接API呼び出しで対応可能（APIキー不要の公開API）
- ジャンル推定は実行時API呼び出しではなく、ビルド前工程（Claude Code）で一括処理する（詳細は5.3節）
- データ更新時のワークフローは10節を参照

---

## 4. 対象ユーザー

個人利用（蔵書所有者のみ）。

## 5. データ要件

### 5.1 入力データ

| 項目 | 内容 |
|------|------|
| 形式 | CSV（Google Apps Script で Google Drive から取得） |
| レコード数 | 約 1,900 行（ファイル単位） |
| 主要カラム | フォルダパス、ファイル名、ファイルサイズ (MB)、MIME タイプ、作成日、最終更新日、ファイルURL、ファイルID |

### 5.2 ファイル名パース

PDF ファイル名は以下の形式に概ね従い、正規表現で段階的に抽出する。

```
[バージョンプレフィックス_]タイトル（シリーズ名）著者名 ページ数p_ISBN.pdf
```

| フィールド | 抽出方法 |
|-----------|---------|
| バージョン | `kindlep_` → Kindle版、`ipad3_` → iPad3版、なし → オリジナル版 |
| ISBN | 末尾の `_XXXXX`（10〜13文字の英数字） |
| ページ数 | 末尾の `NNNp` パターン |
| シリーズ名 | `（）` または `()` 内のテキスト（`／` 区切りで結合） |
| 著者名 | 最後の括弧の後ろに残るテキスト。括弧がない場合は末尾の日本語2単語 |
| タイトル | 上記を除いた残り |

### 5.3 ジャンル分類

フォルダパスベースの分類は採用せず、全書籍について Claude Code による一括推定を行う。

- 対象: 全書籍（819冊）
- 手法: タイトル・著者名をもとに Claude が推定
- 出力: 推定ジャンルを付与した上で `data/books.json` に保存
- 実行タイミング: CSV 処理時（ビルド前工程）。更新頻度が低いため再実行は任意

### 5.4 重複排除

同一書籍の複数バージョンを1件の書籍レコードに統合する。

| 項目 | ルール |
|------|-------|
| 統合キー | ISBN が存在する場合は ISBN、ない場合はパース済みタイトル |
| バージョン | 全バージョンをリストとして保持 |
| ジャンル | より具体的なジャンルを優先（SF > フィクション（日本）> エッセイ > … > 未分類） |
| タイトル・著者 | オリジナル版の情報を優先 |

**除外ファイル**（管理用ファイルのため書籍データから除外）:
- `self_check`
- `20120330_1246_33_0706`
- `to_1733_912_003_5_at`

### 5.5 期待データ量（2026-02-27 時点実績）

| 項目 | 値 |
|------|---|
| 入力 CSV レコード数 | 約 1,900 |
| PDF ファイル数 | 1,889 |
| 重複排除後のユニーク書籍数 | 819 |
| ユニーク著者数 | 366 |
| ジャンル数 | 11（未分類含む） |

## 6. 機能スコープ（概要）

詳細な機能要件は次フェーズで定義する。ここでは提供する機能の範囲を概要レベルで記載する。

| 機能領域 | 概要 |
|---------|------|
| データ処理 | CSV インポート・メタデータ解析・重複排除・ジャンル推定・`data/books.json` 生成 |
| 書籍一覧・検索 | キーワード検索・ジャンルフィルタ・ソート・著者絞り込み・ページネーション |
| 書籍詳細 | 基本情報（ファイル名パース）＋ 補完情報（外部書誌API） |
| 統計・分析 | ジャンル別冊数・著者別ランキング |

**書籍詳細の外部書誌API方針**（決定済み）

| ソース | 役割 | 利用条件 |
|-------|------|---------|
| ファイル名パース | 基本情報（タイトル・著者・ISBN・ページ数・シリーズ）として常に使用 | 不要 |
| OpenBD API | ISBN照合で第一優先（日本語書籍特化） | 無料・APIキー不要 |
| Google Books API | OpenBD でデータなしの場合のフォールバック（洋書対応） | 無料（APIキー推奨） |

ISBNがある書籍は OpenBD → Google Books の順でフォールバック。ISBNがない書籍はファイル名情報のみ表示。

## 10. 制約・前提

| 項目 | 内容 |
|------|------|
| データ取得 | Google Drive の直接 API アクセスは困難なため、Google Apps Script で取得した CSV をインポートする方式を採用する |
| 利用環境 | 個人利用のみ。パブリックアクセスは不要 |
| 外部書誌API | OpenBD・Google Books はAPIキー不要の公開APIのためブラウザから直接呼び出し可能 |

### データ更新ワークフロー

蔵書に追加・変更があった場合の更新手順：

1. Google スプレッドシートで `google_drive_file_lister.gs` を実行し、CSVをダウンロード
2. `data/booklist.csv` を更新
3. Claude Code セッションでデータ処理を実行（パース・重複排除・ジャンル推定）
4. `data/books.json` が更新される
5. SPA をリビルドしてデプロイ
