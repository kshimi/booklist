**作成日**: 2026-02-27
**ステータス**: v1.0 完成（初回構築完了）

## 1. プロジェクト概要

Google Drive に保管しているスキャン済み書籍 PDF（約1,900ファイル）を対象に、ファイル名から書誌メタデータを自動抽出し、検索・閲覧・AIレコメンドが可能な蔵書管理システムを構築するプロジェクト。

### 1.1 背景

- 書籍をスキャンした PDF ファイルが Google Drive の `70_Book` フォルダ配下にジャンル別サブフォルダで保管されている
- ファイル名に一定の規則でタイトル・著者・ページ数・ISBN が埋め込まれている
- 同一書籍に対して、オリジナルスキャン・iPad向け軽量版・Kindle向け軽量版の最大3バージョンが存在する
- 蔵書全体を俯瞰・検索できる仕組みと、読書レコメンドの仕組みが求められた

### 1.2 目標

1. PDF ファイル一覧の自動取得
2. ファイル名からの書誌メタデータ解析（タイトル・著者・ISBN・ページ数・シリーズ・ジャンル）
3. 重複排除による正規化された蔵書カタログの作成
4. Excel形式の蔵書カタログ（オフライン閲覧用）
5. インタラクティブな蔵書ブラウザ（検索・フィルタ・統計・AIレコメンド）

## 3. ツール構成

本プロジェクトは3つの独立したツールで構成される。

### 3.1 ファイルリスト抽出スクリプト（Apps Script）

**ファイル**: `google_drive_file_lister.gs`

Google Drive のフォルダ内容を再帰的に走査し、全ファイルの情報をスプレッドシートに出力する Google Apps Script。

#### 使用手順

1. Google スプレッドシートを新規作成
2. メニュー「拡張機能」→「Apps Script」を開く
3. スクリプトを貼り付けて保存
4. `ROOT_FOLDER_ID` を対象フォルダIDに設定
5. 関数 `listAllFiles` を実行（初回は権限承認が必要）
6. スプレッドシートに結果が出力される → CSV としてダウンロード

#### 出力カラム

| カラム | 説明 |
|--------|------|
| フォルダパス | ルートフォルダからの相対パス（` / ` 区切り） |
| ファイル名 | ファイル名（拡張子込み） |
| ファイルサイズ (MB) | ファイルサイズ（小数第2位） |
| MIME タイプ | `application/pdf` 等 |
| 作成日 | `yyyy-MM-dd HH:mm` 形式 |
| 最終更新日 | `yyyy-MM-dd HH:mm` 形式 |
| ファイルURL | Google Drive 上の直接リンク |
| ファイルID | Google Drive ファイルID |

**ファイル**: `蔵書ブラウザ.jsx`

#### 機能一覧

| 機能 | 説明 |
|------|------|
| 書籍検索 | タイトル・著者名での絞り込み |

- **自由入力**: ユーザー独自の質問を入力可能
- **API エンドポイント**: `https://api.anthropic.com/v1/messages`
- **モデル**: `claude-sonnet-4-20250514`

#### UI デザイン

- ダークテーマベース
- ジャンル別カラーコーディング（ネオンアクセント）
- タブ切替: 一覧 / 統計 / レコメンド
- レスポンシブ対応

## 4. データ処理仕様

### 4.1 ファイル名パース規則

PDF ファイル名は概ね以下の書式に従う:

```
タイトル（シリーズ名）著者名 ページ数p_ISBN.pdf
```

#### パースの処理順序

1. **バージョン判定・除去**: `kindlep_` プレフィックス → Kindle版、`ipad3_` プレフィックス → iPad3版、それ以外 → オリジナル版
2. **拡張子除去**: `.pdf` を除去
3. **ISBN抽出**: 末尾の `_XXXXX` パターン（10〜13文字の英数字）を抽出・除去
4. **ページ数抽出**: 末尾の `NNNp` パターン（半角スペース + 数字 + p）を抽出・除去
5. **シリーズ名抽出**: `（）` または `()` 内のテキストを全て抽出（`／` 区切りで結合）
6. **著者名抽出**:
   - 優先: 最後の括弧の後ろに残るテキストを著者名とする
   - 代替: 括弧がない場合、末尾の日本語2単語パターン（姓 名）にマッチさせる
7. **タイトル**: 上記を除去した残りがタイトル

#### パース例

| ファイル名 | タイトル | シリーズ | 著者 | ページ | ISBN |
|-----------|---------|---------|------|--------|------|
| `「わかる」とは何か （岩波新書） 長尾 真 186p_4004307139.pdf` | 「わかる」とは何か （岩波新書） | 岩波新書 | 長尾 真 | 186 | 4004307139 |
| `kindlep_銀河英雄伝説 田中芳樹 350p_4488725012.pdf` | 銀河英雄伝説 | — | 田中芳樹 | 350 | 4488725012 |

### 4.2 ジャンル分類

フォルダパスのキーワードマッチングによりジャンルを割り当てる。判定は以下の優先順で行い、最初にマッチしたものを採用する。

| 優先度 | キーワード | ジャンル | 備考 |
|--------|-----------|---------|------|
| 1 | `エッセイ` | エッセイ | |
| 2 | `日本の作家` | フィクション（日本） | |
| 3 | ` SF` or `/SF` | SF | 前後にスペースまたはパス区切りが必要 |
| 4 | `ノンフィクション` | ノンフィクション | **フィクションより先に判定**（部分一致回避） |
| 5 | `フィクション` | フィクション | |
| 6 | `実用` | 実用 | |
| 7 | `家庭` / `家庭関係` | 家庭 | |
| 8 | `漫画` / `コミック` | 漫画・コミック | |
| 9 | `コンピュータ` / `O'Reilly` | コンピュータ | |
| 10 | `運転関係` | 運転 | |
| — | マッチなし | 未分類 | |

**重要**: 「ノンフィクション」は「フィクション」の部分文字列であるため、ノンフィクションを先に判定する必要がある。

### 4.3 重複排除（デデュプリケーション）

同一書籍がオリジナル・iPad3・Kindle の最大3バージョン存在するため、以下のロジックで統合する。

#### 統合キー

- ISBN が存在する場合: ISBN をキーとする
- ISBN が存在しない場合: パース済みタイトルをキーとする

#### 統合時の優先ルール

- **バージョン**: 全バージョンをリストとして保持
- **ジャンル**: より具体的なジャンルを優先（下記の優先度テーブル）
- **タイトル・著者**: オリジナル版の情報を優先

#### ジャンル優先度

| 優先度 | ジャンル |
|--------|---------|
| 10 | SF |
| 9 | フィクション（日本） |
| 8 | エッセイ |
| 7 | ノンフィクション、コンピュータ、運転 |
| 6 | 実用、家庭 |
| 5 | 漫画・コミック |
| 4 | フィクション |
| 0 | 未分類 |

この優先度により、たとえば kindle フォルダの SF サブフォルダにある本が、72_BookscanPdf のフィクションフォルダにもある場合、より具体的な「SF」がジャンルとして採用される。

#### 除外ファイル

以下のファイル名はスキャンの管理用ファイルであり、書籍データから除外する:

- `self_check`
- `20120330_1246_33_0706`
- `to_1733_912_003_5_at`

## 7. 開発経緯と設計判断

### 7.1 Google Drive アクセス方式の選定

Google Drive のウェブインターフェースは JavaScript で動的にコンテンツを読み込むため、web fetch では直接ファイル一覧を取得できなかった。Claude からの Google Drive コネクタも利用できなかったため、ユーザーが自身の Google アカウントで Apps Script を実行する方式を採用した。

### 7.2 ファイル名パースの課題

ファイル名の書式には例外が多く、以下のような対応が必要だった:

- 括弧が入れ子になるケース（シリーズ名 + 出版社名の併記）
- 著者名に中黒（・）が含まれるケース（外国人著者のカタカナ表記）
- ISBN が欠落しているファイル
- ページ数表記がないファイル

正規表現による段階的な抽出で、764/819冊（93%）で著者名を、732/819冊（89%）でジャンルを正しく割り当てられた。

### 7.3 ジャンル分類の精度向上

初回の実装では「ノンフィクション」が「フィクション」の部分一致で誤判定される問題と、ISBN による重複排除時にオリジナル版（フィクションフォルダ）の汎用ジャンルが kindle版（SFサブフォルダ）の具体的ジャンルを上書きする問題が発生した。これらはジャンル判定の順序制御とジャンル優先度テーブルの導入で解決した。

### 7.4 データサイズの最適化

フルの書誌 JSON は約287KB だったが、React アーティファクトへの埋め込みに向けてフィールド名の短縮と不要データの除去により約105KBに圧縮した。

## 8. 今後の拡張案

- **ISBN からの書誌情報補完**: OpenBD や Google Books API を使い、出版社・出版年・表紙画像などを取得
