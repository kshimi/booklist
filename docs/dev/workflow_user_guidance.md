# 開発操作ガイダンス

Claude Code を活用した日常開発の操作手順です。

## 開発フロー概要

```
1. Issue 作成        ← 開発者
2. 設計書作成指示     ← 開発者 → Claude Code
3. 設計書レビュー     ← 開発者
4. 実装指示          ← 開発者 → Claude Code
5. コードレビュー     ← 開発者
6. マージ            ← 開発者
```

## 1. Issue の作成

GitHub で Issue を作成します。テンプレートが用意されています。

### バグ報告の場合

- 「Bug Report」テンプレートを選択
- バグの内容、再現手順、期待する動作を記入

### 機能要望の場合

- 「Feature Request」テンプレートを選択
- 概要、背景・動機、実現イメージを記入

### その他

- ブランクIssue で自由に記述可能

### Issue 作成のポイント

- タイトルは簡潔かつ具体的に
- 本文に要件・受け入れ条件を明記する
- 関連する仕様ドキュメントがあればリンクを記載する

## 2. Claude Code への設計指示

Issue を作成したら、Claude Code に設計書の作成を指示します。

```
#<Issue番号> の設計書を作成してください
```

または design-doc スキルを直接呼び出します。

```
/design-doc #<Issue番号>
```

### Claude Code が行うこと

1. GitHub Issue の内容を読み込み
2. 既存コードの調査
3. 設計書の作成（`docs/app/design/` 配下）
4. Issue へのコメント投稿（設計書リンク）

## 3. 設計書のレビュー

Claude Code が作成した設計書をレビューします。

### レビューの観点

- 要件を満たしているか
- 技術的アプローチは妥当か
- 影響範囲は適切に把握されているか
- テスト計画は十分か

### 修正が必要な場合

Claude Code に修正を指示します。

```
設計書の「技術的アプローチ」セクションを修正してください。〇〇の方式ではなく△△を採用してください。
```

### 承認

設計書に問題がなければ、Claude Code に承認を伝えます。

```
設計書を承認します。詳細仕様を更新してください。
```

Claude Code は `docs/app/spec/` の仕様を更新します。

## 4. 実装の指示

仕様更新が完了したら、実装を指示します。

```
#<Issue番号> を実装してください
```

または implement スキルを直接呼び出します。

```
/implement #<Issue番号>
```

### Claude Code が行うこと

1. ブランチの作成
2. 実装
3. テスト実行
4. コミット
5. PR 作成

## 5. コードレビュー

Claude Code が作成した PR をレビューします。

### レビュー方法

GitHub の PR ページで確認するか、Claude Code にレビューを依頼できます。

```
PR #<PR番号> をレビューしてください
```

または review スキルを使用します。

```
/review #<PR番号>
```

### レビューの観点

- コードの品質・可読性
- 設計書との整合性
- テストの十分性
- セキュリティ上の問題がないか

### 修正が必要な場合

修正点を Claude Code に伝えます。

```
PR のレビューコメントに対応してください
```

## 6. マージとクリーンアップ

レビューが完了したら、以下を実施します。

### マージ

GitHub の PR ページで「Merge pull request」を実行します。

### Issue のクローズ

PR に `Closes #<Issue番号>` が含まれていれば自動的にクローズされます。
含まれていない場合は手動でクローズします。

### ブランチの削除

マージ後、不要になったブランチを削除します。

```bash
# リモートブランチの削除（GitHub PR ページからも可能）
git push origin --delete <branch-name>

# ローカルブランチの削除
git branch -d <branch-name>
```

## UIのレビューポイント

UIに関わるPRをレビューする際は、開発サーバーを起動して以下の動作を確認してください。

```bash
npm run dev
```

### 書籍一覧（S-1）の動作確認

| 確認項目 | 操作 | 期待する結果 |
|---------|------|-------------|
| キーワード検索 | 検索バーにテキスト入力 | 入力と同時に絞り込まれる |
| ジャンルフィルタ | 大ジャンルのボタンをクリック | そのジャンルの書籍のみ表示、サブジャンル行が展開される |
| サブジャンルフィルタ | サブジャンルボタンをクリック | 大ジャンルとの AND 条件でさらに絞り込まれる |
| 著者絞り込み | 著者ドロップダウンから選択 | その著者の書籍のみ表示される |
| ソート | ソートドロップダウンを切り替え | 一覧の並び順が変わる |
| ページネーション | ページ番号またはページ切替ボタンをクリック | 該当ページが表示される |
| フィルタリセット | 「すべて」をクリック | 全件表示に戻る（ページも1ページ目に戻る） |
| 複合条件 | 検索＋ジャンル＋著者を組み合わせ | すべての条件が AND で適用される |

### 実装済み画面

| 画面 | 状態 | 主な機能 |
|------|------|---------|
| 書籍一覧（S-1） | 実装済み | 検索・フィルタ・ソート・ページネーション |
| 書籍詳細（S-2） | 実装済み | 基本情報表示・Google Drive リンク・外部書誌情報（書影・出版社・あらすじ） |
| 統計ダッシュボード（S-3） | 実装済み | ジャンル別統計・著者別ランキング |

## データメンテナンス

### 著者名補正のメンテナンス

`data/book-corrections.json` に著者名を手動で追記することで、自動パターンでは分離できなかった書籍の著者名を補完できます。

#### 手順

**1. 未登録書籍を確認する**

```bash
node scripts/list-missing-authors.js
```

著者名が空で、かつ `book-corrections.json` に未登録の書籍一覧が表示されます。

**2. `data/book-corrections.json` に著者名を記入する**

該当エントリの `author` フィールドに正しい著者名を記入します。
タイトルに著者名が混入している場合は `title` フィールドも修正してください。

```json
{
  "original_title": "GNU Emacs デブラ キャメロン",
  "title": "GNU Emacs",
  "author": "デブラ・キャメロン"
}
```

**3. `books.json` を再生成する**

```bash
node scripts/process.js
```

**4. 変更を確認してコミットする**

```bash
git diff data/books.json
git add data/book-corrections.json data/books.json
git commit -m "chore: add author corrections for <書籍名など>"
```

#### 注意事項

- `book-corrections.json` の変更は `books.json` の再生成まで反映されない
- 著者名の表記は `data/author-aliases.json` のエイリアス照合後に上書きされるため、エイリアス登録済みの表記とは独立して機能する
- 著者名が本当に不明な書籍は `author: ""` のまま残してかまわない

### 書誌情報エンリッチメント

`scripts/enrich.js` を実行することで、外部APIから書影・出版社・出版年・あらすじを取得し、
`data/book-metadata.json` に保存します。書籍詳細画面で即時表示されるようになります。

#### 実行タイミング

- `process.js` 実行後（新しい書籍が追加されたとき）
- 書誌情報の取得漏れを補完したいとき

#### 手順

**1. 書誌情報を取得する**

```bash
node scripts/enrich.js
```

差分取得のため、すでに取得済みのISBNは自動でスキップされます。
実行完了後、API別のヒット数とスキップ数がサマリー表示されます。

**2. SPA を起動して確認する**

```bash
npm run dev
```

書籍詳細を開き、書影・あらすじが表示されることを確認します。

#### オプション

| オプション | 説明 |
|-----------|------|
| `--force` | 全件を再取得する（取得済みデータも上書き） |
| `--skip-google` | Google Books API をスキップする |

#### 注意事項

- `data/book-metadata.json` は **Git管理対象**です。`enrich.js` 実行後にコミットしてください
- `book-metadata.json` がない状態でも SPA は動作します（書籍詳細開時にランタイムAPIコールにフォールバック）

## ヒント

### Claude Code への効果的な指示の出し方

- Issue 番号を必ず伝える
- 曖昧な表現を避け、具体的に指示する
- 大きなタスクは Issue を分割して段階的に進める

### よく使うコマンド

| 操作 | コマンド |
|------|---------|
| 設計書作成 | `/design-doc #<番号>` |
| 実装 | `/implement #<番号>` |
| コミット | `/commit` |
| レビュー | `/review #<PR番号>` |
