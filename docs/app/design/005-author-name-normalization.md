# 設計ドキュメント 005: 著者の名寄せ

**Issue**: #17 著者の名寄せを行う
**作成日**: 2026-03-01
**ステータス**: レビュー待ち

---

## 概要

元データ（`data/booklist.csv`）に存在する著者名の表記ゆれを吸収し、同一著者を一貫した名称で扱えるようにする。

**表記ゆれの例（Issue より）:**
- `アイザック アシモフ` ↔ `アイザック・アシモフ`
- `J．R．R． トールキン` ↔ `J．R．R．トールキン`

現状、`scripts/process.js` のパース処理（F-1）はファイル名からそのまま著者名を抽出するため、上記のような揺れがそのまま `books.json` に反映される。結果として、同一著者が複数の著者名として登録され、統計画面の著者ランキングや著者絞り込み検索に影響が出る。

---

## 技術的アプローチ

### アプローチ比較

| # | 方式 | 概要 | メリット | デメリット |
|---|------|------|---------|-----------|
| **A** | **正規化ルール方式** | process.js の F-1 内で、著者名に対して一定の変換ルールを適用して正規化する | 追加ファイル不要、ルールが明示的 | ルールで吸収できない個別の揺れへの対応が難しい |
| **B** | **エイリアステーブル方式** | `data/author-aliases.json` に表記ゆれ→正規名のマッピングを定義し、process.js が参照して置換する | 個別の揺れに完全対応できる、管理が容易 | 全件目視確認が必要、ファイル管理コストがかかる |
| **C** | **組み合わせ方式（推奨）** | 正規化ルールで汎用的な揺れを吸収した上で、残った個別ケースをエイリアステーブルで補完する | 両方の利点を持つ | 実装が2段階になる |

### 推奨: アプローチ C（組み合わせ方式）

**理由:**
- 中黒・スペース・全角ドットなどの汎用的な表記ゆれはルールで機械的に吸収できる
- 実名の綴り違いなど個別ケースはルールで対処困難なため、エイリアステーブルで補完する
- 段階的な適用により、著者名の変換ロジックが透明で保守しやすい

---

### 実装詳細（アプローチ C）

#### ステップ 1: 正規化ルール（F-1 パース時に適用）

| ルール | 変換内容 | 例 |
|--------|---------|-----|
| 中黒統一 | 氏名中の半角スペース（` `）を中黒（`・`）に置換 | `アイザック アシモフ` → `アイザック・アシモフ` |
| 全角ドット統一 | `．` を `·`（中黒）またはピリオド（`.`）に統一（イニシャル表記向け） | `J．R．R．` → `J.R.R.` |
| 前後のホワイトスペース除去 | trim 処理（現行でも実施済み） | — |

> **注意**: 中黒への変換は著者名が2単語の場合（姓 名の形式）に限定し、誤変換を防ぐ。

#### ステップ 2: エイリアステーブル（正規化後に適用）

**ファイル**: `data/author-aliases.json`

```json
{
  "J.R.R. トールキン": "J.R.R.トールキン",
  "J. R. R. トールキン": "J.R.R.トールキン"
}
```

- キー: 揺れ表記（正規化ルール適用後の文字列）
- 値: 正規名（`books.json` に格納する名前）
- 正規名はテーブルの値側にも存在できる（自己参照は不要）

#### 処理フロー（変更後）

```
F-1: ファイル名パース
    │
    ▼ 著者名抽出（現行通り）
    │
    ▼ 正規化ルール適用（新規: スペース→中黒、全角ドット統一）
    │
    ▼ エイリアステーブル照合（新規: author-aliases.json 参照）
    │
    ▼ 正規化済み著者名を出力
    │
    ▼ F-2〜F-4 へ（変更なし）
```

#### 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `scripts/process.js` | F-1 パース処理に正規化ルールとエイリアス照合を追加 |
| `data/author-aliases.json` | 新規作成（実データ確認後に内容を確定） |
| `data/books.json` | `process.js` 再実行により再生成 |
| `docs/app/spec/functional/data-processing.md` | F-1 仕様に著者名正規化の記述を追加 |
| `docs/app/spec/system/data-pipeline.md` | ステップ2に正規化処理の詳細を追加 |

## テスト計画

### 単体テスト（`scripts/process.test.js`）

| テストケース | 期待結果 |
|------------|---------|
| `アイザック アシモフ` を正規化する | `アイザック・アシモフ` |
| `J．R．R． トールキン` を正規化する | エイリアステーブルを経て `J.R.R.トールキン` |
| エイリアステーブルに存在しない著者名は変更しない | 正規化ルール適用後の名前をそのまま返す |
| エイリアステーブルが存在しない場合でも処理が続行される | エラーにならず、ルール正規化のみ適用 |

### 手動確認

1. `node scripts/process.js` を実行して `books.json` を再生成
2. 統計ダッシュボード（S-3）の著者ランキングで、名寄せ前に分かれていた著者が統合されていることを確認
3. 著者絞り込み検索で正規名・揺れ表記どちらで検索しても同一著者の書籍が表示されることを確認（フロントエンド動作確認）
4. `jq '[.[].author] | unique | sort | .[]' data/books.json` でユニーク著者一覧を確認し、明らかな重複がないことを目視確認
