# 設計ドキュメント 006: 書名・著者名分離の改善

**Issue**: #19 書名と著者名の分離ができていない書籍がある
**作成日**: 2026-03-02
**ステータス**: レビュー待ち

---

## 概要

現在の `scripts/process.js` における F-1 パース処理は、ファイル名中の**括弧（`（）` / `()`）の後ろに続くテキストを著者名**として抽出する。

しかし、括弧を持たないファイル名（例: `GNU Emacs デブラ キャメロン.pdf`）では著者名の抽出が行われず、タイトルに著者名が混入したままになる。

**現状（`data/books.json` 調査結果）:**

| 項目 | 件数 |
|------|------|
| 総書籍数 | 819 |
| 著者名が空の書籍数 | **81**（約 10%） |

**問題の例（Issue より）:**

| ファイル名（括弧なし） | 期待されるタイトル | 期待される著者名 |
|--------------------|----------------|---------------|
| `GNU Emacs デブラ キャメロン.pdf` | GNU Emacs | デブラ・キャメロン |
| `SQLアンチパターン Bill Karwin.pdf` | SQLアンチパターン | Bill Karwin |
| `Graham - Knuth - Patashnik - Concrete Mathematics.pdf` | Concrete Mathematics | Graham / Knuth / Patashnik |
| `iPhone SDK アプリケーション開発ガイド Jonathan Zdziarski.pdf` | iPhone SDK アプリケーション開発ガイド | Jonathan Zdziarski |

---

## 問題の分析

### 著者名が空になるケースの分類

81 件の著者名が空の書籍を分析した結果、以下のパターンに分類できる。

| # | パターン | 件数（概算） | 自動化難度 | 例 |
|---|---------|------------|----------|-----|
| P1 | 漢字氏名（姓 名）がタイトル末尾 | 約23件 | 中 | `Schemeによる記号処理入門 猪股 俊光` |
| P2 | カタカナ氏名がタイトル末尾 | 約5件 | 中 | `GNU Emacs デブラ キャメロン` |
| P3 | 欧文氏名が日本語タイトルの末尾 | 約10件 | 中 | `SQLアンチパターン Bill Karwin` |
| P4 | 著/訳マーカーを含む | 約2件 | 高 | `科学は不確かだ！ R．P．ファインマン 著 大貫 昌子 訳` |
| P5 | 著者名なし（ファイル名に著者情報なし） | 約10件 | — | `Rubyベストプラクティス`、`アンコール・ワットを読む` |
| P6 | その他・複合・判定困難 | 約31件 | 低 | `Graham - Knuth - Patashnik - Concrete Mathematics`、`ファインマンの手紙 リチャード P ファインマン` |

### 判定困難なケースの具体例

| ファイル名（タイトルフィールド） | 困難な理由 |
|------------------------------|-----------|
| `Graham - Knuth - Patashnik - Concrete Mathematics` | 著者名がタイトルより前にある（逆順フォーマット） |
| `Joel on Software Joel Spolsky` | 全欧文でタイトルと著者名の境界が不明確 |
| `指輪物語〈追補編〉 新板 J．R．R． トールキン` | `〈〉`括弧は `（）` でないため現在の処理でスキップ |
| `ファインマンの手紙 リチャード P ファインマン` | イニシャル混在・スペース区切り |
| `ビーイング・デジタル - ビットの時代 新装版 ニコラス・ネグロポンテ` | 中黒で連結されたカタカナ名（スペース区切りでない） |
| `最高齢プロフェッショナルの教え 徳間書店取材班` | `取材班`は著者名ではなく編集部門 |

## 技術的アプローチ

### アプローチ比較

| # | 方式 | 概要 | メリット | デメリット |
|---|------|------|---------|-----------|
| **A** | **自動パターン検出** | 正規表現でタイトル末尾の著者名パターンを自動抽出 | 全自動、追加作業なし | 偽陽性リスクあり（タイトルの一部を著者名と誤認する恐れ） |
| **B** | **手動補正ファイル** | `data/book-corrections.json` に手動でタイトル・著者名の正解を記載 | 正確、全ケースに対応 | 81件の手動作業が必要 |
| **C** | **ハイブリッド（推奨）** | 高確度な自動検出パターンで一部を自動処理し、残りを補正ファイルで手動補完 | 手動作業を削減しつつ精度を確保 | 実装が2段階になる |

---

### 推奨: アプローチ C（ハイブリッド方式）

**理由:**
- パターン P1〜P3（約38件）は一定の確度で自動検出できるが、偽陽性リスクを排除しきれないため、検出結果をユーザーが確認できる仕組みが必要
- P5（著者情報なし）・P6（判定困難）は自動化できない（約41件）
- データの正確性を最優先するため、`data/book-corrections.json` を正解データとして機能させ、自動検出は**あくまで補助**とする

---

### 実装詳細（アプローチ C）

#### ステップ 1: 補正ファイルによる手動定義（最優先）

**ファイル**: `data/book-corrections.json`

```json
{
  "corrections": [
    {
      "original_title": "GNU Emacs デブラ キャメロン",
      "title": "GNU Emacs",
      "author": "デブラ・キャメロン"
    },
    {
      "original_title": "Rubyベストプラクティス",
      "title": "Rubyベストプラクティス",
      "author": "Jeremy McAnally"
    }
  ]
}
```

- `original_title`: 現在 `books.json` に記録されているタイトル（著者名が混入した状態）。一致キーとして使用
- `title`: 正しい書名
- `author`: 正しい著者名（空文字列可）
- `process.js` が F-1 パース後にこのファイルを照合し、一致した書籍のタイトル・著者名を上書きする

#### ステップ 2: 補助的な自動パターン検出

括弧なしのファイル名に対して、以下の高確度パターンのみを自動適用する。
いずれのパターンにも一致しない場合は、補正ファイルへの登録待ちとして著者名を空のまま残す。

**自動検出パターン（`parseFilename` 内で適用）**

| パターン | 条件 | 変換例 |
|---------|------|-------|
| P-A: 欧文名（日本語タイトル末尾） | タイトルに日本語が含まれ、末尾に `[A-Z][a-z]+ [A-Z][a-z]+` または `[A-Z][a-z]+` が続く | `SQLアンチパターン Bill Karwin` → タイトル: `SQLアンチパターン` / 著者: `Bill Karwin` |
| P-B: 漢字氏名（2単語で末尾） | タイトルが `[\u4E00-\u9FFF]{1,4} [\u4E00-\u9FFF]{1,4}` で終わる（誤検知を避けるため長さ制限を設ける） | `Schemeによる記号処理入門 猪股 俊光` → タイトル: `Schemeによる記号処理入門` / 著者: `猪股 俊光` |

> **注意**: カタカナ氏名（P2）や著/訳マーカー（P4）は誤検知リスクが高く（タイトル末尾のカタカナ語との区別が困難）、自動パターンには含めない。これらは補正ファイルで対応する。

#### ステップ 3: 未解決書籍リスト出力スクリプト

**ファイル**: `scripts/list-missing-authors.js`

```bash
node scripts/list-missing-authors.js
```

- `data/books.json` を読み込み、著者名が空の書籍を一覧出力する
- `data/book-corrections.json` に登録済みの書籍は除外する
- ユーザーが補正ファイルの記入作業を管理するためのヘルパーツール

#### 処理フロー（変更後）

```
F-1: ファイル名パース
    │
    ▼ バージョン・ISBN・ページ数・シリーズ名抽出（現行通り）
    │
    ▼ 括弧後ろの著者名抽出（現行通り）
    │
    ▼ [新規] 著者名が空の場合のみ: 自動パターン P-A, P-B を試行
    │
    ▼ 著者名正規化・エイリアス照合（#17 実装済み）
    │
    ▼ [新規] book-corrections.json との照合・上書き（最後に適用）
    │
    ▼ F-2〜F-4 へ（変更なし）
```

## テスト計画

### 単体テスト（`scripts/process.test.js`）

| テストケース | 期待結果 |
|------------|---------|
| `SQLアンチパターン Bill Karwin.pdf` をパースする（P-A パターン） | `title: "SQLアンチパターン"`, `author: "Bill Karwin"` |
| `Schemeによる記号処理入門 猪股 俊光.pdf` をパースする（P-B パターン） | `title: "Schemeによる記号処理入門"`, `author: "猪股 俊光"` |
| `book-corrections.json` に登録済みのタイトルが上書きされる | 補正ファイルの値が優先される |
| `book-corrections.json` が存在しない場合でも処理が続行される | エラーにならない |
| 自動パターンにも補正ファイルにも一致しない書籍は著者名が空のまま | 著者名 `""` のまま出力される |

### 手動確認

1. `node scripts/list-missing-authors.js` を実行し、著者名が空の書籍リストを確認
2. `data/book-corrections.json` を記入後、`node scripts/process.js` を実行
3. `jq '[.[].author] | group_by(.) | map({author: .[0], count: length}) | sort_by(.count) | reverse | .[:10]' data/books.json` で著者ランキングを確認
4. `books.json` 中の著者名が空の件数が減少していることを確認
5. SPA の著者絞り込み検索で、補正後の著者名で書籍が正しく表示されることを確認

## 作業フロー（実装後のメンテナンス）

補正ファイルへの登録は以下のワークフローで段階的に進める:

1. `node scripts/list-missing-authors.js` を実行して未登録書籍一覧を確認
2. 正しいタイトル・著者名を調査して `data/book-corrections.json` に追記
3. `node scripts/process.js` を実行して `data/books.json` を再生成
4. 変更を確認・コミット

## 未決事項

- `data/book-corrections.json` の初期登録は実装と別作業（81 件分を手動で記入）
- 自動パターン P-A の「日本語が含まれる」判定をどの閾値で行うかは実装フェーズで確定（例: ひらがな・漢字が1文字以上あること）
- P-B パターンの漢字氏名の長さ制限（`{1,4}` など）は実データでの偽陽性率を確認して調整
- `〈〉` 括弧（全角山括弧）を `（）` と同様に処理対象とするかは実データで確認後に判断
